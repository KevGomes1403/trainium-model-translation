Qwen3.6-35B-A3B — Multi-Token-Prediction (MTP) head

A visual companion to the implementation plan. The MTP head is a self-speculative drafter: one extra decoder layer that reads the main model's hidden state plus the next token's embedding, and guesses one more token ahead. The main model then verifies that guess in a single batched step.

The idea in one line

Normal decoding makes 1 token per forward pass. MTP makes 2 — then checks.

Each decode round the main model emits token xt+1 as usual. The MTP head cheaply drafts xt+2 by reusing the hidden state the main model already computed. A verify pass confirms or rejects the draft. If accepted, you advanced 2 tokens for ~1.x the cost of one — that's the speedup. Greedy output is bit-identical to non-MTP decoding (verification guarantees it).

Notation. hj is the hidden after reading the token at position j; the main head turns it into the next token xj+1 (so hj → xj+1). The MTP head turns (hj, embed(xj+1)) into a draft of xj+2. Sn = the DeltaNet state after the verify block's first n tokens.

The only change to the main model
Today the traced forward() returns logits + KV cache + DeltaNet state. We add one extra output: the hidden state before model.norm (the amber tensor above). For prefill that's the last position; for the 2-token verify block it's the hidden at every position (shape [B, n_active, H]) — that's how the host gets both ht+1 and ht+2. It's appended after the existing outputs so KV/DeltaNet alias offsets don't move. Everything else stays bit-for-bit the same.

Why this predicts t+2

ht already encodes everything through position t. The main head turns it into xt+1. The MTP head is given that answer (xt+1) plus the rich hidden state, and is trained to jump one further — to xt+2. One MTP layer ⇒ one extra token per round.

Diagram 3 · one decode round (k = 1 draft)

Carry → Draft → Verify → Select

We always carry three things from prefill or the previous round: the real token xt+1, the hidden ht that produced it, and the committed DeltaNet/KV state after xt.

1 · Draft cheap: 1 layer

xt+2draft = MTP(ht, embed(xt+1)) — reuse the hidden we already have to guess one token past the real one.

2 · Verify 1 backbone pass · 2 positions

Run the backbone over the block [xt+1, xt+2draft], seeded from the committed state. It emits — at each position — a hidden, a next-token logit, and a DeltaNet state checkpoint:
pos t+1 · read xt+1 → ht+1 → predicts xt+2true · save S1 pos t+2 · read xt+2draft → ht+2 → predicts xt+3 · save S2

The MTP head runs after Select — never before
Its two inputs are both chosen by the outcome: the hidden (ht+2 on accept, ht+1 on reject) and the token to embed (xt+3 vs xt+2true). One rule covers both: draft from the hidden at the last accepted position, embedding the new real token. Drafting before the decision would read a hidden that only exists if the draft was right.

There is no separate single-token step: the verify pass is the next round's main step. Committed tokens are exactly what plain greedy decoding would produce — speculation changes only the speed: 1 or 2 tokens per backbone pass.

Diagram 4 · the one tricky part

The DeltaNet state can't be sliced — so we pick a saved checkpoint

30 of 40 layers are Gated DeltaNet, a linear recurrence: each token folds irreversibly into a state (a recurrent matrix S plus a small conv window). Unlike a KV cache you can truncate, you cannot peel the draft token back out of S.

Fix: emit a checkpoint per position, select by accept count
The verify pass already steps the recurrence once per block position — so have it save the state after each: S1 (after the real token) and S2 (after the draft). The recurrence runs forward only; "rollback" is just choosing Saccept_count — S2 on accept, S1 on reject. No inversion, no recompute. The conv window, GQA KV cache, and MTP KV cache are truncated to the same accept count.

This is exactly what mature stacks do for the same Qwen3-Next-style architecture: vLLM (gdn_attn — one state slot per speculative position, indexed by num_accepted_tokens) and SGLang (MambaRadixCache state "sandboxes").

Why not invert the recurrence?

Algebraically S1 is recoverable from S2, but the gate is exp(cumulative log-decay), which underflows hard on this checkpoint (the same overflow that forced the chunked DeltaNet kernel over the fused one). Dividing it back out is numerically unsafe — and no production stack does it. Forward-only + checkpoint-select is the standard.

Why it's cheap

At k=1 there are only two candidates per layer, so "select" is a Python index, not a gather. One extra state per DeltaNet layer ≈ 1 MiB; both candidates across all 30 layers ≈ 60 MiB. The block is 2 tokens, so the verify recurrence is two exact single-step updates — sidestepping the chunked-decay overflow entirely.

NxDI's own enable_fused_spec can't be reused: it assumes a separate draft model with its own sliceable KV cache, and the hybrid cache manager explicitly rejects speculation today. So we trace the MTP head as its own small graph and orchestrate draft/verify on the host.

Diagram 5 · why not one fused graph?

The destination is one fused launch — the recurrence is what makes it hard here

Fusing the backbone and the MTP head into a single device launch (no host round-trips) is the right end state. The blocker on this model isn't host overhead — it's that a fused launch must perform two operations inside a static, ahead-of-time-compiled graph that are trivial for a KV cache and genuinely hard for a recurrence.

The cost model: only one number matters

A backbone pass (40 layers) is ~45 ms here (~22 tok/s). The MTP head (1 layer) is ~1/40 of that. So speed = backbone passes per round:

Schemebackbone passes / roundtokens / roundresult
Plain decoding111 tok / pass
Fuse main+MTP → 2 tokens, then a separate full-model verify 21–2 ≤ 1 tok / pass — slower than plain
Backbone-over-block (verify + seed) + tiny MTP head 11–2 up to 2 tok / pass

A separate verify graph is a second full-model forward — you pay for the 40 layers twice. The win comes from making the verify pass double as the next round's main step, which is why the head is the small separable piece, not the verifier.

The recurrence tax (paid whether you fuse or not)

10 GQA layers — easy
"Keep only the accepted prefix" = scatter K/V for the accepted positions into an addressable cache; ignore the rest. A runtime index into a fixed-size buffer. Standard.
30 DeltaNet layers — hard
The state is a recurrence: S₂ = f(f(S_c, x_{t+1}), x_{t+2}ᵈ). The block irreversibly folds the draft into the state — you can't slice it off. On reject you need "state after 1 token," so you must compute a candidate end-state for each accept count and select by mask, for all 30 layers.

What fusing adds on top of that tax

Fused single launch — all in the static graph
  • Dynamic accept/commit as masked selects (graphs are static-shape; no if accepted)
  • Candidate-state select on [B,H,128,128] × 30 DeltaNet layers
  • Reworks the one-transition-per-launch state aliasing (recurrent/conv_state_buffer)
  • Two sub-forwards (block backbone @ n_active=2, MTP head @ n_active=1) in one graph
  • NxDI's fused-spec assumes sliceable KV caches; _validate_hybrid_config raises on spec
Hot-loop = 2 graphs + host glue (prefill is a 3rd, run once)
verify backbone (block): logits + per-position hiddens + S₁,S₂
── host ── accept_count, select S_accept, truncate caches
MTP head: selected hidden + embed → next draft

Both graphs are clean & static; the if accepted / index math lives in plain Python between launches. The verify graph reuses the existing chunked-prefill-from-committed-state path at n_active = 2.

ROI — why we build two graphs first
Fusing the two launches into one saves a single tiny kernel dispatch (tens of µs) out of a ~45 ms round — about 1–2%. The hard recurrence-candidate-state logic is needed either way. So: ship the two-graph + host loop, prove greedy-equivalence and measure acceptance, then fuse for the last ~1–2%. That's the hard 20%, done last, for the smallest slice of the win.
