Derived shapes the kernels actually use

None of these appear in the config file — they fall out of it, and they are the numbers you will see hardcoded as module constants.

# DeltaNet, per rank
key_dim    = Hk · d        =  4 · 128 =  512
value_dim  = Hv · d        =  8 · 128 = 1024
conv_dim   = 2·key_dim + value_dim    = 2048   # q, k, and v are all convolved
I          = conv_dim + value_dim + 2·Hv = 3088   # fused in_proj width (+ a, b gate scalars)
rep        = Hv / Hk       = 2                    # GQA expansion inside DeltaNet
state_w    = K_conv − 1    = 3                    # conv sliding-window depth

# GQA, per rank
N          = 4 Q + 1 K + 1 V heads    = 6          # the fused qkv head axis
I_DIM      = N · D         =  6 · 256 = 1536       # fused qkv width
GATE_DIM   = 4 · D                    = 1024
VALUE_DIM  = 4 · D                    = 1024       # o_proj contraction
D_TILES    = D / 128                  = 2          # the head_dim=256 tiling factor
N_SUB      = 4 · D_TILES              = 8          # o_proj sub-heads of 128
rope_dim   = 0.25 · D                 = 64

# Hardware / layout
H0 = nl.tile_size.pmax    = 128       # SBUF partitions — a hardware constant
H1 = H / H0               =  16       # free-axis width of the residual
H2 = H1 / n_prgs          =   8       # per-LNC-core free width (tp2013)
T  = B · S                =   2       # verify pass: 1 committed + 1 speculative

The KV-head subtlety

The model has 2 KV heads but is deployed at TP=4, so KV heads are replicated, not split — each rank gets 1. The true model-wide GQA ratio is 16:2 = 8:1, but the ratio a kernel sees is 4:1. Read NUM_Q_HEADS=4, NUM_KV_HEADS=1 in fused_layer.py with that in mind.

The layer pattern

full_attention_interval = 4 puts a GQA layer at every index ≡ 3 (mod 4). The megakernel receives this as the layer_is_gqa boolean list, which is also the cache key for the generated flat signature:

Two collisions to watch for

I means the fused in_proj width (3088) in DeltaNet and the expert intermediate size (128) in MoE. K means the conv kernel width (4) in DeltaNet and the router top-k (8) in MoE. Both are reused in the source with no disambiguation — context is the only guide.

The shape of it

Three composables, called in a Python for loop that unrolls at trace time. Everything else is plumbing around them.

attention_layer_compose

DeltaNet (linear attention). Input norm → fused in_proj → causal conv → gated delta-rule recurrence → gated RMSNorm → o_proj. Runs on 30 layers.

Shards on value heads, per LNC core

gqa_fused_compose

Grouped-query attention with head_dim=256. Input norm → qkv+gate projections → qk-norm → partial RoPE → attention → sigmoid gate → o_proj. Runs on 10 layers.

Shards on s_prior, then H for o_proj

moe_layer_compose

The whole MoE FFN. Norm once → router top-8 → routed experts + shared expert → σ-gate → gated sum. Runs on all 40 layers.

Shards on tokens, per LNC core

The one-line summary

All of the math that is specific to Qwen3.6 — the DeltaNet recurrence, the MoE expert loop, qk-norm, partial RoPE — is hand-written NKI in this repo. Only three things come from outside: the two bookend GEMM kernels (qkv_tkg, output_projection_tkg) and rmsnorm_tkg, all called directly from nkilib; plus two forks — attention_tkg (patched to lift the head_dim ≤ 128 cap) and router_topk (patched to accept a per-layer name prefix).

The residual spine

One SBUF tile lives for the entire kernel. Every layer reads it, and adds two reduced partials back into it. Nothing else persists.

# qwen36_verify_megakernel.py:177 — the whole kernel, abridged
residual = ndarray[H0=128, T*H1] sbuf              # 8 KB, tp2013, identical on both cores
_load_residual_to_sbuf(residual, X)                  # ← the only HBM read of hidden state

for i in range(40):
    x_sb = residual.reshape((H0, T, H1))           # free reshape, no copy

    if layer_is_gqa[i]:
        attn_partial, active_k, active_v = gqa_fused_compose(x_sb, ..., out_in_sb=True)
    else:

    residual += _all_reduce_gather_h(attn_partial)     # TP all-reduce + LNC gather on H

return (output, *gqa_active_kv, *dn_candidates)      # every mutated handle, or DCE eats it

What still touches HBM inside the loop

The residual doesn't, but three things do, by design: weights stream in per layer (~22.6 MiB per core per layer — two layers will not fit, so there is no cross-layer weight ring); DeltaNet candidate states are written out per layer; and GQA active K/V are written out per layer. The last two are outputs the host needs for the accept/reject commit.

Memory & layout

One layout convention, tp2013, is what lets attention and MoE share a single residual tile with no repacking. It is worth ten minutes before reading any kernel.

The index map

With H=2048, H0=128 partitions, H1=16 free, n_s = n_prgs = 2 LNC cores, H2 = H1/n_s = 8:

# SBUF free index f, partition index h0  →  logical hidden column h
s, h2 = divmod(f, H2)
h     = s*(H0*H2) + h0*H2 + h2          # tp2013

# at n_s == 1 this degenerates EXACTLY to tp102:
h     = h0*H1 + f

The property that matters: an H-contraction shard is a contiguous free-slice. That is what makes the attention H-gather and the MoE token-gather both cheap on the same tile.

Trap

tp2013 is derived from n_prgs and is LNC-coupled — never hardcode it. Separately, rmsnorm_tkg picks its H-layout from lnc, not from the token-shard decision, which is why NORM_SINGLE_CORE_FORCED = False in the MoE path. At T=1 with 2 cores the two conventions used to disagree silently; that is now an assert.

Stale docstrings — do not trust these two

moe/__init__.py and specs/moe_layer.md still describe the routed experts as nkilib moe_tkg and the shared expert as mlp_tkg. Neither is imported anywhere on the live path. Both were replaced by hand-written raw NKI — routed to fix a DMA fragmentation problem, shared because mlp_tkg's weight loader hardcodes a tp102 reshape that cannot express tp2013. The only surviving moe_tkg reference is the torch oracle in tests/.

LNC sharding: value heads

Each core gets three disjoint channel segments — its slice of q, its slice of k, its slice of v. Conv and recurrence then run as a self-contained per-core pipeline with zero cross-core traffic. GQA expansion stays local because Hv = rep · Hk holds per core.

The one cross-core exchange

o_proj contracts over all value heads, but each core holds only its own. The fix is a single sendrecv of the small attention tensor before the matmul — then output_projection_tkg H-shards the large output. No reduce of the result is needed, because the H-shards are disjoint.

Two different RMSNorms — don't confuse them

gamma/eps is the input norm, fused inside qkv_tkg. z_gamma/z_eps is the gated per-head norm at the recurrence output seam: out = x·rsqrt(mean(x²)+eps)·gamma·silu(z). The gated norm affects only attn_out — it never touches the SSM state or the candidates.

Reader's trap: attn_shape in attention_layer_compose looks like a live tensor but never receives data. It exists purely so the recurrence can read T and dim off its shape on the SBUF path.

Why norm and RoPE come before the transpose

With head_dim on the free axis, D = 256 fits under the 512-wide free limit, so RMSNorm over D is a single free-axis reduce per token — no tiling at all. A head_dim-on-partition norm would have capped d at 128. The single free→partition transpose then happens once, after RoPE, and folds the 1/√D Q scale into a copy it had to do anyway.

MoE path

The whole FFN as one SBUF composable: normalize once, then four consumers read that same tile. Runs on all 40 layers, and is the largest DMA consumer in the kernel.

Token sharding — and the garbage columns

Each LNC core owns a token slice and computes the full H for it. There is no H-split of MoE work, and no cross-core exchange inside the kernel.

The reason the megakernel calls moe_tkg_shard_decision a second time

sigma_gate_compose and gated_sum write all T columns, but routed_local/shared_local are only valid on this core's slice. The foreign-token columns are garbage-but-populated, not stale. The megakernel recomputes (T_offset, T_len) at line 274 purely so it can slice out the valid block before the all-reduce. Never read outside it.

The DMA fix that made this path viable

Stock moe_tkg loads each expert's gate and up weights as two separate strided [H, I] views, so the innermost contiguous run is only I elements. That fragments the load into tens of thousands of tiny packets — sync-engine bound, tensor engine starved. The hand-written replacement loads gate+up as one contiguous [H0, H1, 2I] slab and slices it in SBUF, with a 2-slot prefetch ring so expert k+1's DMA overlaps expert k's compute.

Measured: 129 µs → 71 µs, sync-engine share 72% → 13%. The kernel became tensor-bound.

Note: the module docstring describes the run as 256 B → 512 B; a later audit of the actual access pattern found both forms coalesce to 4 KB. The 129→71 µs result is the reliable number.

Why that extra copy is not redundant

Tokens live on the free axis (f = t·H1 + h1), so a token block is a contiguous free-slice — but the slice inherits the parent's partition stride (T*H1) while spanning only T_len*H1. That is a partition-strided view, and nccl's SBUF collectives require a densely packed 2-D operand. Feeding the sub-view directly is precisely the bug that produced garbage output; the tensor_copy into a fresh tile is the fix.

nccl constraints worth memorizing

  • SBUF operands must be exactly 2-D with a contiguous free dim. No PSUM, no HBM/SBUF mixing.
  • One tensor per SBUF call; op ∈ {add, minimum, maximum}.
  • sendrecv and core_barrier are ISA ops (nki.isa), not part of nki.collectives.
  • replica_groups is a traced constant — a plain List[List[int]], no torch.distributed in the path.

State & cache writes

This is a speculative verify kernel: it processes T candidate tokens, but not all will be accepted. So neither attention type commits its state. Both emit per-position candidates and let the host pick one.

The unifying idea

Every state output carries a leading T axis indexed by position = accept count. On a reject the host selects index accept_count − 1. No inversion, no recompute — the rollback is a pointer select. DeltaNet's recurrent state and conv window are candidate-tracked in lockstep, because a reject must roll back both.

The window after token t is a pure slice, win[:, t+1 : t+1+state_w], of the packed [conv_state | qkv] window. Each column is transposed and packed, then one DMA per token scatters the segment into conv_cand[t].

The property that makes this safe: the candidate columns are exact copies of the raw projection output, not of the conv output. A committed state is therefore bit-identical to what a fresh forward pass would have carried.

Why every mutated handle is returned

The kernel returns (output, *active_k/v, *cand_state/conv_cand) — 81 tensors at N=40. This is not convenience: if a mutated buffer is not returned, NCC dead-stores the write. The model then re-associates them per layer so NxDI aliases the mutated HBM buffers rather than the pre-kernel ones.

Design decisions

The choices that shaped the code, and what was rejected. Several rejections are load-bearing — they explain why obvious-looking optimizations are absent.

SBUF-resident residual

The residual is 8 KB and lives in SBUF for all 40 layers. Only the entry load and final store touch HBM, killing ~120 hidden-state round-trips.

Enabled by out_in_sb / output_in_sbuf flags added to all three composables — default-off, so the standalone HBM contracts stayed byte-identical.

tp2013 as the shared layout

One index map, parameterized on n_prgs, that degenerates exactly to tp102 at LNC=1 — so both core counts share one code path with no branch. Attention and MoE read the same residual tile with no repacking.

Cost accepted +2.9% on the standalone MoE layer, and it forecloses an unsplit 8 KB gate/up load — which the 2-way pipelining split had already given up.

MoE token-shards; attention H-shards

Deliberately decoupled: the work split and the SBUF H-layout are two independent decisions. MoE keeps token-sharding and gets a token-axis gather.

Rejected: shard MoE on H. The byte math is exactly break-even at T=2 (12 MiB/core either way), it buys zero bandwidth, and it would cost 32 cross-core sendrecvs per layer plus a re-derivation of the fused slab DMA.

Raw-NKI routed experts

Replaced nkilib moe_tkg to fix DMA fragmentation: one contiguous [H0,H1,2I] gate+up slab plus a 2-slot prefetch ring. 129 µs → 71 µs.

Contract preserved exactly — top-K, SiLU, POST_SCALE by the already-L1-normalized affinity with no re-normalization — so the torch moe_tkg oracle still validates it.

Raw-NKI shared expert

nkilib's mlp_tkg gate/up loader hardcodes a tp102 reshape with no layout parameter, so a core cannot address a full 16-wide tp2013 tile.

Rejected: patch the vendored view. It's one SwiGLU and the machinery already existed in routed_experts_nki. (The public wrapper is also broken for the SBUF + NO_NORM case — a second, independent reason.)

Fork attention_tkg, don't rewrite it

Lifting head_dim ≤ 128 touched three tiling sites; the entire bf16 compute path stayed byte-for-byte AWS code, with each region delimited by explicit patch markers.

Also kept a from-scratch reference core (attention_fresh_ref.py) off the live path, purely for cross-checking.

Name-prefix vendoring

Tracing a subkernel 40× collides on op names and will not compile. Most nkilib kernels accept a caller-supplied sbm and de-collide for free — but router_topk had no prefix hook at all, which made it the one hard blocker.

Vendored for this reason alone — the fork adds a name_prefix kwarg threaded to 7 explicit op-name sites. Not a math change.

exec()-generated flat signature

NKI's frontend classifies tuple/list top-level args as scalars — no HBM binding, and aliasing breaks. So every weight and cache must be its own positional arg: ~675 of them at N=40, code-generated and cached per layer_is_gqa pattern.

Also rejected: stacked-by-layer tensors. The outer-dim stride trips NCC_IBIR158/243 out-of-bounds.

No cross-layer weight ring

One layer is ~22.6 MiB per core; two would be ~45 MiB against ~30 MiB usable. Weights stream per layer, and only intra-layer overlap is available.

Consequence grouping layers is free — there is no ring to lose — which is why the build could de-risk at N=4 before scaling to 40 at zero perf cost.

Candidates in HBM, not SBUF

Holding 30 layers' candidate states in SBUF would be 7.5–15 MiB/core and would collide with the streaming weights. They are written to HBM per layer.

Rejected: router_w pre-permute. The predicted descriptor split was real and exactly as forecast, but measured at −0.02 µs — inside the noise. Per-packet DMA throughput already saturates at 4 KB.

Where the win actually comes from

Not FLOPs. It is 120 kernel launches → 1, 80 XLA all-reduces → in-kernel collectives that overlap with compute, and ~120 HBM round-trips of the hidden state removed.

Watch for in-kernel all-reduce cost rising per layer — prior art saw 13.45 → 27.77 µs — and still netting a large win because sync-engine and DMA time fell further.

Not decorated with nki.jit

The body function is plain Python; build_verify_megakernel wraps the generated flat signature at the call site. Decorating both would double-jit and overflow the stack.

Practical effect the kernel is traced through a cached, code-generated wrapper keyed on the layer-type tuple.

Reading guide

Suggested order for a first pass through the code, given everything above.

  1. The shell first. qwen36_verify_megakernel.py is only ~380 lines and the layer loop is 100 of them. Read it top to bottom, ignoring the composable bodies — you are reading for the residual lifecycle and the two gathers.
  2. Then the layout. Convince yourself of the tp2013 index map. Nearly every access pattern in the MoE kernels is an expression of it, and they are unreadable until it clicks.
  3. Then one composable, end to end. moe_layer.py is the best first choice — it's the shortest, and its norm-once → four-consumers structure is the clearest example of the SBUF-resident pattern.
  4. Then the state paths. recurrence.py::_write_state and the GQA active_k/active_v emission. Both are small, and they explain why the return tuple is 81 tensors long.
  5. Leave the vendored files for last. attention_tkg.py is ~4000 lines of AWS code with ~40 lines of patch. Grep D256 PATCH and read only those regions — the rest is not ours and does not need to be understood to work on this kernel.

Two docstrings that will mislead you

  • moe/__init__.py and specs/moe_layer.md claim the routed experts use moe_tkg and the shared expert uses mlp_tkg. Both are wrong — the live code is raw NKI in this repo.
  • moe_layer.py says "No cross-rank all-reduce here (megakernel/model boundary)". True of the composable, but the megakernel has since crossed that boundary and calls nccl.all_reduce itself.

Naming that isn't what it sounds like

  • moe_tkg_shard_decision is a vestigial name — it wraps a 15-line pure-Python function in this repo and calls nothing from moe_tkg.
  • attn_shape in the DeltaNet composable is a shape carrier, never a live buffer.
  • affinities passed into router_topk is never written — the router rebinds its output internally. Consume the returned tensors.
