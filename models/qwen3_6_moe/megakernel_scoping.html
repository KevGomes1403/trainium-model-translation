0 · TL;DR

Feasible, and smaller than it sounds. One speculation round (spec_len=2, k=1) is 1 draft layer + 40 verify layers (×2 tokens) + 2 LM-head passes ≈ ~41 fused layer instances — the same order as the 48-layer Qwen3-30B megakernel that already ships at 1.76× over XLA. The hard part is not scale; it is heterogeneity and in-kernel control flow: DeltaNet recurrence with candidate-state checkpointing, a 248k-vocab argmax across TP ranks inside the kernel, a data-dependent embedding gather (draft token → verify input), and a greedy accept/commit select — none of which appear in the existing kernels.

Headroom: rounds currently run at 9.4 ms p50 (186 tok/s decode). A bandwidth roofline on the per-round weight traffic (~2 GB/rank) puts the floor near 2–3 ms. Realistic target: ~4–5 ms/round (≈ 350–450 tok/s) after Phase 3.

Plan: 5 phases. The first two — DeltaNet decode block in NKI, then a T=2 hybrid verify megakernel with the draft/accept/commit still in XLA — deliver most of the win at a fraction of the risk. Whole-round fusion (in-kernel argmax, embedding gather, accept, state select) lands in Phases 3–4.

1 · First principles: why decode is DMA-bound and why a megakernel helps

Three facts drive everything in single-stream decode at bs=1 on Trainium:

1.1 Arithmetic intensity ≈ 1 flop/byte

Every weight matrix used at decode is touched by exactly one (or here, two–three) token rows. A matmul [1, H] @ [H, N] does 2·H·N flops over 2·H·N weight bytes (bf16) — ratio ≈ 1. Trainium3's PE arrays need hundreds of flops/byte to saturate. So the decode floor is HBM bandwidth, not compute:

round_time_floor = bytes_streamed_per_round / sustained_HBM_BW

The XLA baseline pays additionally for: per-graph launch/sync, fragmented small DMAs, HBM round-trips of activations between fused regions, and serialized layers. The whole megakernel value proposition is to take bytes_streamed to its model-defined minimum and overlap 100% of compute under the DMA stream — exactly the win seen in your profiles: 43% fewer DMA packets, 2.7× larger packets, hidden-state HBM writes 10× lower (gpt-oss: 0.14 MB vs 1.4 MB/token).

1.2 SBUF as the layer-to-layer register file

The residual [B, S, H] at H=2048, S≤2, bf16 is ≈ 8 KB — trivially SBUF-resident for the whole round. That is the structural trick of ~/nki-moe/megakernels/qwen3_moe/transformer_qwen.py:160 (residual_sb lives in SBUF across all 48 layers). New here: DeltaNet recurrent state per rank is [B, 8, 128, 128] bf16 = 256 KB/layer × 30 layers ≈ 7.7 MB (modeling_qwen36_a3b.py:376), which fits in SBUF (~24 MB/core) — meaning live + candidate states can stay on-chip for the entire round and only the committed state needs an HBM write.

1.3 What changes with speculation

Speculation converts 1 round = 1 token (avg 1.76 here) into 1 round = ~1.76 tokens for ~3 tokens of compute — profitable because the verify (2 tokens) streams weights once. The XLA fused-spec graph already exploits this; the megakernel's job is to remove the remaining sync/DMA tax across draft + 40 verify layers + LM heads + state commit, all chained inside one NEFF.

3 · Prior art you already own

Qwen3-30B-A3B (48L, TP=4, LNC=2)

  • ~/nki-moe/megakernels/qwen3_moe/transformer_qwen.py:103–436 — full decode step in one nki.jit; SBUF-resident residual; double-buffered weight ring; hoisted per-layer norms/RoPE (:178–227); SB2SB all-reduce + LNC gather.
  • T>1 speculative variant exists — transformer_qwen3_moe_speculative.py:49–53: per-slot RoPE [d/2,B,T], causal verify mask, consecutive KV scatter from base position. This is the direct skeleton for the verify trunk.
  • 8.15 ms vs 14.37 ms XLA (1.76×).

GPT-OSS-20B (24L, TP=8) + vendored nkilib

  • Alternating mask schedules per layer (transformer_gpt_oss.py:309–364) — direct precedent for the 3:1 DeltaNet/GQA alternation.
  • Vendored, validated subkernels: attention_block_tkg, router_topk (top-k over E in-kernel), moe_tkg + selective_expert_impl with index-driven indirect weight DMA (~/nki-moe/nki_kernels/moe/selective_expert_impl.py) — the same primitive the embedding gather and selective experts need here.
  • Lesson learned: HBM round-trips between subkernels (MXFP4 path) eat the win — keep everything SBUF-in/SBUF-out.

Integration pattern is proven: layer 0 dispatches the kernel, layers 1..L−1 pass through, KV aliased in-place (qwen_with_megakernel.py:434–486). Reusable verbatim under Qwen36FusedSpecModel.

5 · Target design: one round, one kernel

qwen36_round_megakernel(hidden_seed, last_token_id, embed_table, mtp_weights,
                        per-layer {Wqkvz/Wo, conv_w, A_log/dt_bias, router, gate_up, down, norms},
                        K/V caches[10+1], deltanet S/conv buffers[30], lm_head_T, cos/sin, pos_id):
  # ① DRAFT  — embed(last_token)‖hidden_seed → eh_proj → GQA layer → norm → lm_head ⊗ x
  draft_logits → TP-global argmax → t̂  (idx + offset all-gather, 4 ranks)
  # ② EMBED  — indirect DMA: embed_table[t̂] → SBUF (selective_expert-style gather)
  # ③ VERIFY — 40 layers, T=2 [t₀, t̂]; DeltaNet: seed→step→S₁(keep)→step→S₂(keep)
  #            GQA: scatter K/V @ pos, pos+1; MoE: router_topk T=2, selective experts
  # ④ HEADS  — lm_head over 2 positions → argmax ×2 → target tokens
  # ⑤ ACCEPT — accept = (t̂ == target₀);  S* = select(accept, S₂, S₁) per layer (SBUF)
  # ⑥ WRITE  — S*, conv* → HBM state buffers; chosen hidden → rolling buffer slot
  return tokens(2), accept_idx   # host: position_ids += 1+accept, mask update only

The host's per-round work drops to incrementing positions and the attention mask — identical to the contract NxDI's fused-spec runtime already exposes (model_base.py:2799).

The draft-replay stage (③ in today's graph) can stay in the kernel as one extra GQA layer over T=2, or remain in XLA in early phases; for k=1, drafting from the chosen hidden next round is equivalent — to be confirmed against the test_mtp_state_rule.py oracle before any deletion.

6 · Key challenges & blockers, ranked

C1DeltaNet decode block in NKI with candidate states

Per token, per head: decay S·e^{g}, read k·S, rank-1 update S += kᵀδ, read q·S on a [128,128] state (modeling_qwen36_a3b.py:405–424) — vectorE-heavy, near-zero tensorE use; plus L2-norm, gating (exp·softplus), and the causal conv (4-tap, window [2560,3]). Existing NKI kernels (nki_kernels/nki_deltanet.py) are prefill-shaped (per-(B,H), full sequence). Needed: a TKG-shaped fused block — projections + conv + 2 steps + checkpoint S₁/S₂ in SBUF — with the bf16 carry rounding (:441–444) preserved or it breaks bit-exactness with the oracle. Largest single line item; no precedent.

C2In-kernel LM head + TP-global argmax over 248,320 vocab

Two passes/round at 254 MB/rank ≈ ~36% of the round's HBM floor. Argmax over 62k local logits is a cheap vectorE reduction; the global step is a 4-rank all-gather of (max, idx) + offset compare — router_topk proves top-k in NKI, nccl.all_reduce/sb2sb exists in your kernels. Risk is plumbing, not concept. Sampled (non-greedy) decode is explicitly out of scope (greedy round is what ships today, :3667–3672).

C3Token-dependent embedding gather inside the kernel

Verify needs embed[t̂] where t̂ is computed in ②. With untied embeddings (table 248320×2048, row 4 KB) this is one indirect HWDGE descriptor keyed by an SBUF index — the exact mechanism of selective_expert_impl.py's router-indexed weight loads. Sequential dependency adds latency but it's one row.

C4Conditional commit & KV semantics on reject

State commit is a per-layer select between SBUF-resident S₁/S₂ — replicating :3675–3689 without HBM bounce (the XLA graph already proves select semantics; never strided views). GQA KV needs no rollback: write both slots, mask the rejected one next round, as fused spec does today. Draft KV equivalence on reject is the subtle bit — gate Phase 4 on the CPU oracle (tests/test_mtp_state_rule.py).

C5GQA specifics: head_dim 256, partial mRoPE

head_dim 256 > PMAX 128 (your nkilib fork handles it) and partial rotary 64/256 with interleaved mRoPE [11,11,10] (:1389–1396) must be reproduced in the attention subkernel — table-driven cos/sin keeps it simple. Per-slot RoPE/masks for T=2 are solved (transformer_qwen3_moe_speculative.py:49–53).

C6MoE at E=256/K=8 + shared expert; compile/aliasing scale

Same selective machinery at half the per-rank intermediate (I=128); legacy-load constraint I%128==0 holds. Shared expert + three sigmoid gates are minor vectorE additions. ~145 aliased outputs and the exec()-generated signature need extension, plus fp32 router softmax with normalized top-k (:1410–1413). Tedious, low risk; budget compile-time wrestling.

Profile & pin baseline — ~2 days

DeltaNet TKG block in NKI (C1) — 2–3 wks · critical path

Standalone: projections + conv + 2 recurrent steps + S₁/S₂ checkpoint, SBUF-in/out, bf16 carry. Validate vs verify_block per-(B,H) bit-exact. Exit: green vs CPU oracle.
2

T=2 verify trunk megakernel — 2–3 wks

Fork the speculative skeleton; alternate [Δ,Δ,Δ,GQA]; head_dim-256 attention + mRoPE; E=256 MoE + shared expert; candidates/hidden out. Draft, LM heads, accept, commit stay XLA. Exit: token-identical; verify ≲5 ms.
3

Heads in: draft + LM heads + argmax + embed gather (C2, C3) — 2 wks

Two sequential megakernel calls; activations HBM-handed once each. Exit: ≤6 ms round.
4

Whole-round fusion (C4) — 1–2 wks

Accept + select + state write in-kernel; collapse to one NEFF; host = pos/mask only. Gate on oracle. Exit: ≈4–5 ms, ~350–450 tok/s.
5

Optimize

Weight ring across heterogeneous layers, LNC=2 DeltaNet head split, LM-head dominance (quantized head / shortlist), bs>1.
